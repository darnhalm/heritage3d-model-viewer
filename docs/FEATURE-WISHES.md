# Пожелания по новым функциям

Документ для накопления идей и пожеланий по новым возможностям Model Viewer. Сюда можно дописывать пункты по мере появления.

---

## Список пожеланий

### Подхват файла настроек при одновременном перетаскивании с моделью

**Описание:** Если пользователь перетаскивает в окно и модель, и файл настроек (`*.model-viewer-settings.json`) вместе (с тем же базовым именем, что и модель, или любой такой файл), автоматически читать содержимое файла настроек и применять его после загрузки модели, не показывая файл настроек в списке сцены.

**Контекст:** Реализация вызывала зависание лоадера (прогресс-бар останавливался примерно в середине), поэтому поддержка перетаскивания модели и настроек вместе отключена. Автоподхват настроек **из папки модели по URL** (когда модель открыта по ссылке и рядом лежит файл с тем же именем) работает.

**Желательно при доработке:** обеспечить стабильную загрузку без зависаний при перетаскивании модели и файла настроек вместе (таймауты, исключение файла настроек из списка ресурсов лоадера и т.п.).

### Цветокоррекция и кастомный LUT для сплатов

**Описание:** Добавить возможность применять кастомный LUT (Look-Up Table) для цветокоррекции Gaussian Splatting-моделей. Сейчас tonemap реализован аналитическими формулами (ACES, Filmic, Hejl и т.д.) в шейдере; LUT даёт более гибкий контроль и кинематографический вид.

**Контекст:** Готовых библиотек под PlayCanvas нет. Можно адаптировать логику из `glsl-lut` (npm) — GLSL-функция для 2D LUT 512×512, framework-agnostic. LUT практически не утяжеляет рендер (одна texture fetch на пиксель). Нужен post-process pass с full-screen quad и UI для выбора/загрузки текстуры LUT.

### Поведение параметра `?load=` при разных сценариях развёртывания

**Описание:** Сейчас значение `?load=` приводится к абсолютному URL (`new URL(loadUrl, window.location.href).href`) перед загрузкой модели. Решить в будущем, как обрабатывать относительные пути и сценарии развёртывания.

**Контекст:**
- При деплое в подпуть (например, `/viewer/`) путь `?load=/models/scull.glb` превращается в `http://site.com/models/scull.glb`, а не `http://site.com/viewer/models/scull.glb`. Если модели лежат относительно приложения, загрузка может не сработать.
- Если появится загрузка модели из файла конфигурации (URL в JSON), нужно определить: пути относительно origin, относительно приложения (base href) или относительно самого конфиг-файла.
- Текущая конвертация не затрагивает `model-viewer-settings.json` и `localStorage` — они используют отдельный код.

**Желательно при доработке:** добавить учёт base href / базового URL приложения; при необходимости — сделать режим конвертации опциональным.

### Выбор качества текстур при загрузке модели

**Описание:** Добавить выбор качества текстур (`Auto / High / Medium / Low`) при загрузке модели, чтобы viewer мог открывать одну и ту же модель с разным разрешением текстур в зависимости от устройства или выбора пользователя.

**Контекст:**
- Сейчас, если модель приходит с `8K` текстурами, viewer загружает именно их. Обычные GPU mipmaps не решают задачу выбора качества загрузки: `8K` всё равно сначала скачивается и попадает в память.
- Полноценного runtime texture streaming / texture LOD, как у крупных платформ вроде Sketchfab, в проекте сейчас нет.
- Наиболее реалистичная реализация для этого viewer — не downscale уже загруженной текстуры, а подготовка нескольких наборов текстур заранее (`8K / 4K / 2K / 1K`) и выбор нужного набора в момент загрузки.
- Для `.gltf` это проще через подмену путей к текстурам. Для `.glb` практичнее либо хранить несколько отдельных версий модели, либо иметь preprocessing pipeline, который извлекает и готовит texture tiers заранее.

**Желательно при доработке:**
- добавить в UI `Texture Quality: Auto / High / Medium / Low`;
- реализовать выбор через заранее подготовленные texture sets, а не через runtime downscale;
- в режиме `Auto` выбирать уровень по типу устройства, VRAM/производительности и/или размеру viewport;
- при возможности предусмотреть совместимость с `KTX2` и внешним asset pipeline.

### Привязка material overrides к конкретному объекту, а не только к имени материала

**Описание:** Перевести сохранение пользовательских изменений material channels (`Diffuse Color`, `Specular Color`, `Metallic`, `Roughness`, `Opacity`) с ключа по `material.name` на более точный object-scoped ключ, например `nodePath + materialIndex`.

**Контекст:**
- Сейчас overrides сохраняются в `model-viewer-settings.json` по имени материала. Для большинства сцен этого достаточно.
- Если в модели несколько разных объектов используют материалы с одинаковым именем, один и тот же override может примениться сразу ко всем совпавшим материалам.
- Для текущего этапа это не критично, но при росте числа object-specific правок ограничение станет заметнее.

**Желательно при доработке:**
- хранить overrides по стабильному ключу конкретного объекта и material slot;
- при импорте настроек применять их к точному object/material assignment, а не ко всем материалам с одинаковым именем;
- по возможности сохранить обратную совместимость с уже экспортированными JSON, где ключом служит `material.name`.

### Управление POI из внешнего сайта при embed-встраивании viewer

**Описание:** Добавить возможность управлять точками интереса (`POI`) из внешней страницы, если viewer встроен на сайт как embed/iframe. Это позволит размещать в тексте статьи или карточки объекта ссылки и кнопки, которые активируют соответствующую точку интереса внутри встроенного viewer.

**Контекст:**
- Сейчас `POI` существуют только внутри самого viewer и управляются через локальный UI.
- Для музейных и контентных страниц типичный сценарий — в тексте есть ссылка вроде «см. точку 3», и по клику viewer должен:
  - активировать нужный `POI`,
  - показать/закрепить label,
  - при необходимости сфокусировать камеру на точке.
- Так как `POI` уже можно переупорядочивать, внешний API лучше строить не на отображаемом номере, а на стабильном `id`.

**Желательно при доработке:**
- добавить embed API через `postMessage` между внешней страницей и iframe viewer;
- поддержать команды вроде:
  - `focus-poi`
  - `open-poi`
  - `clear-poi`
- предусмотреть fallback через URL-параметр (`?poi=...`) для прямых ссылок и initial state;
- по возможности добавить обратное событие из viewer наружу (`poi-selected`), чтобы внешний сайт мог подсвечивать активный фрагмент текста;
- строить API на стабильном `poi.id`, а не на визуальном номере точки.

### Контурные аннотации и зоны на поверхности модели

**Описание:** Добавить инструмент аннотаций на поверхности модели не только в виде одиночных `POI`, но и в виде контуров/зон. Базовый сценарий: пользователь ставит несколько точек на поверхности, а viewer соединяет их линией или сглаженной кривой, привязанной к модели.

**Контекст:**
- Для задач реставрации, фиксации повреждений и визуального анализа часто нужна не отдельная точка, а область: трещина, скол, загрязнение, участок реставрации, граница интересующей зоны.
- Полноценная “Bezier-кривая на поверхности” или залитая область на меше — заметно более сложная задача, чем `POI`. Реалистичный первый этап — surface-anchored path: набор точек, поставленных raycast’ом на поверхность, соединённых линией или сглаженным контуром.
- Дальнейшее развитие может включать:
  - незамкнутые paths,
  - замкнутые контуры,
  - полупрозрачные залитые области на поверхности,
  - редактирование контрольных точек по аналогии с `POI`.

**Желательно при доработке:**
- начать с MVP в виде surface path / closed contour;
- использовать тот же подход surface anchoring, что и у `POI` и измерений (`raycast + point + normal`);
- хранить аннотации в settings JSON как отдельную сущность (например, `annotations`);
- предусмотреть цвет, название и список контрольных точек;
- только после этого рассматривать более сложную задачу заливки области на поверхности.

### Ручная подстройка масштаба AR поверх калибровки сцены

**Описание:** Добавить отдельный AR multiplier, который работает поверх уже откалиброванного реального масштаба сцены. Это позволит сохранить физически корректную базу от `measure.unitScale`, но при необходимости чуть увеличить или уменьшить модель в AR для удобства показа.

**Контекст:**
- Сейчас AR уже использует `measure.unitScale`, то есть после калибровки сцены размер модели в AR привязан к реальному масштабу.
- Для демонстрационных и музейных сценариев этого часто недостаточно: иногда нужно слегка увеличить объект для удобства просмотра, не теряя при этом “правильную” исходную базу.
- Полностью отказываться от калиброванного масштаба ради фиксированного коэффициента не стоит: это ломает физическую интерпретацию размеров в AR.

**Желательно при доработке:**
- добавить отдельный AR multiplier поверх `measure.unitScale`;
- хранить multiplier независимо от общей калибровки сцены;
- использовать формулу вида `arScale = measure.unitScale * userArMultiplier`;
- предусмотреть reset multiplier к значению `1.0`;
- сохранить текущую логику: базовый реальный масштаб должен определяться именно калибровкой сцены, а не случайным AR-коэффициентом.
