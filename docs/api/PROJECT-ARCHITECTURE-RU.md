# Архитектура проекта Heritage3D

Этот документ описывает общую архитектуру проекта: что относится к порталу Heritage3D, что относится к серверу хранения/плеерам, где живут данные и почему между слоями нужен API.

## 1. Главный принцип

В проекте нужно разделять две разные части:

```text
Портал Heritage3D = культурные объекты, H3DID, каталог, роли, метаданные, публикация
Storage/player = файлы, обработка, хранение, превью, manifest, embed, просмотр
```

Портал не должен превращаться в файловое хранилище, а сервер хранения/плеера не должен знать всю музейную карточку объекта. Иначе API быстро смешает бизнес-логику, культурное описание, файлы, права и viewer-настройки в одну тяжелую систему.

## 2. Сквозная идентификация

Главный ID проекта — `H3DID`, он же `h3druObjectId`.

```text
H3DID / h3druObjectId -> twinId -> h3druAssetId -> storageAssetId
```

| Сущность | Где живет | Зачем нужна |
|---|---|---|
| `H3DID / h3druObjectId` | Портал Heritage3D | Сквозной ID культурного объекта. Используется в карточке, каталоге, CIDOC/Dublin Core, коллекциях и внешних связях. |
| `twinId` | Портал Heritage3D | ID конкретного цифрового представления объекта: PlayCanvas 3D, Geoscan Cloud, gigapixel 2D, point cloud и т.д. |
| `h3druAssetId` | Портал Heritage3D | Запись ассета на стороне сайта: тип, статус, связь с двойником, доступность скачивания, связь со storage. |
| `storageAssetId` | Storage/player | Технический ID файла или набора файлов в хранилище. Нужен для upload, download, preview, manifest и обработки. |

Важно: `H3DID` и `twinId` не одно и то же. Один культурный объект может иметь несколько цифровых представлений.

`E42 Identifier` в CIDOC CRM не заменяет и не дублирует `H3DID`. Это класс CIDOC для описания идентификаторов. Поэтому `H3DID` остается главным системным ID портала, а при экспорте/описании в CIDOC CRM он может быть записан как один из `E42 Identifier` вместе с инвентарным номером, КАМИС, Госкаталогом, ЕГРОКН и другими внешними идентификаторами.

```text
H3DID = главный ID в системе Heritage3D
E42 Identifier = CIDOC-представление идентификатора в семантической модели
```

## 3. Портал Heritage3D

Портал — это главный бизнес-слой системы. Он отвечает за то, что видит пользователь как сайт: каталог, карточки объектов, коллекции, организации, рабочие группы, роли, модерацию и публикацию.

Основные зоны ответственности портала:

| Блок | Что хранит / делает |
|---|---|
| Объекты | `H3DID`, тип объекта, название, статус, владелец, коллекции, публикация. |
| Метаданные | Dublin Core, CIDOC CRM, профили описания, внешние идентификаторы: КАМИС, Госкаталог, ЕГРОКН, Geoscan Cloud UUID. |
| Организации | Музеи/учреждения как владельцы объектов и коллекций. |
| Рабочие группы | Пользователи внутри организации: редакторы, модераторы, администраторы. |
| Коллекции | Смысловые группы объектов, включая вложенные коллекции через `parentCollectionId`. |
| Публикация | Черновики, модерация, публичный доступ, доступ по паролю, архив. |
| Интеграция со storage | Создание загрузки, получение статуса, получение превью, получение embed, получение download-ссылок. |

Фронтенд сайта должен работать с API портала, а не напрямую с БД и не напрямую со storage/player.

## 4. Storage/player

Storage/player — это технический слой. Он не хранит полную культурную карточку объекта и не решает, кто имеет право публиковать объект в каталоге. Он получает уже проверенные серверные запросы от портала.

Основные зоны ответственности storage/player:

| Блок | Что делает |
|---|---|
| Загрузка файлов | Выдает upload URL, принимает файлы, проверяет лимиты и типы файлов. |
| Хранение | Хранит исходники, производные файлы, оптимизированные модели, gigapixel tiles, preview. |
| Обработка | Запускает конвертацию, оптимизацию, генерацию превью, manifest, производных форматов. |
| Технические metadata | Размер файла, формат, checksum, количество полигонов, bounding box, статус обработки, ошибки. |
| Manifest | Отдает техническое описание того, что должен загрузить viewer. |
| Embed | Формирует `iframeUrl`, `iframeHtml`, query-параметры и краткоживущий `embedToken`, если нужен. |
| Download | Выдает временные ссылки на скачивание, если портал разрешил скачивание. |
| Webhooks | Сообщает порталу, что обработка завершилась, файл готов, возникла ошибка или превышен лимит. |

Пример: если у организации базовый тариф 50 GB и место закончилось, storage/player возвращает техническую ошибку `quota_exceeded`, а портал показывает пользователю понятное сообщение.

## 5. Где хранятся данные

Рекомендуемая схема хранения:

```text
MySQL/PostgreSQL
  - пользователи портала и связи с Geoscan Login
  - организации, рабочие группы, роли
  - H3DID, объекты, коллекции, статусы
  - Dublin Core metadata
  - CIDOC CRM nodes/relations или JSON-профили
  - связи h3druAssetId -> storageAssetId

S3 / object storage
  - исходные файлы
  - оптимизированные модели
  - текстуры
  - gigapixel tiles
  - preview / screenshots
  - manifest-файлы
  - архивные копии

Storage/player database
  - storageAssetId
  - upload sessions
  - processing jobs
  - file technical metadata
  - storage paths
  - download/embed tokens
```

Для MVP можно хранить Dublin Core как плоские поля или JSON в БД портала. Для CIDOC CRM лучше предусмотреть графовую структуру `nodes` и `relations`, даже если физически она сначала будет лежать в MySQL/PostgreSQL.

## 6. Почему нельзя работать напрямую с БД

API нужен обязательно. Прямая работа фронтенда с БД плоха по нескольким причинам:

| Причина | Что сломается без API |
|---|---|
| Безопасность | Пользователь сможет обойти серверную проверку ролей, статусов и владельца объекта. |
| Модерация | Черновики, review, published, password access и archive должны проверяться централизованно. |
| Интеграция со storage | Загрузка файла — это не просто insert в БД, а upload URL, лимиты, обработка, webhooks, статусы. |
| Аудит | Нужно знать, кто создал, изменил, опубликовал или удалил объект. |
| Валидация | Dublin Core, CIDOC CRM, профили описания и права должны проверяться до записи. |
| Совместимость | API можно менять версионно, а прямой доступ к БД привязывает фронтенд к внутренней структуре таблиц. |

Правильная схема:

```text
Frontend сайта -> API портала -> БД портала
API портала -> Storage/player API -> S3 / processing / viewer
Viewer iframe -> Storage/player manifest/embed API
Viewer iframe -> window.postMessage -> Страница сайта
```

## 7. Авторизация и права

Geoscan Login отвечает на вопрос: кто пользователь.

Heritage3D отвечает на вопрос: что пользователь может делать внутри портала.

Storage/player доверяет не пользователю напрямую, а порталу. То есть опасные действия, например загрузка, удаление, публикация, скачивание исходников, идут через API портала. Портал проверяет роли и только после этого вызывает storage/player API своим сервисным токеном.

Рекомендуемая модель:

| Уровень | Кто вызывает | Для чего |
|---|---|---|
| Пользовательский токен | Браузер -> API портала | Личный кабинет, создание объектов, формы metadata, коллекции, модерация. |
| Сервисный токен портала | API портала -> Storage/player API | Загрузка, удаление, preview, manifest, embed, download. |
| Embed token | Публичная страница/iframe -> Viewer | Краткоживущий доступ к конкретному опубликованному представлению. |

Даже если объект публичный и iframe постоянно отображается на сайте, токен на embed или download лучше ограничивать по сроку. Это защищает от копирования прямых ссылок, доступа к черновикам и случайной публикации временных ресурсов.

## 8. Статусы и доступ

На уровне портала объект и цифровой двойник должны иметь статус:

| Статус | Значение |
|---|---|
| `draft` | Черновик, виден владельцу/редакторам/админам. |
| `review` | Отправлен на модерацию. |
| `approved` | Одобрен, но еще не опубликован. |
| `published` | Публично доступен в каталоге. |
| `password_protected` | Доступен по ссылке и паролю/токену. |
| `archived` | Снят с публикации, но сохранен. |
| `deleted` | Удален логически или ожидает физического удаления. |

Storage/player может иметь свои технические статусы:

| Статус | Значение |
|---|---|
| `uploading` | Файл загружается. |
| `queued` | Файл ожидает обработки. |
| `processing` | Идет конвертация/оптимизация. |
| `ready` | Файл готов к просмотру. |
| `failed` | Ошибка обработки. |
| `archived` | Файл перемещен в архивное/ледяное хранение. |

Публичность объекта решает портал. Готовность файла решает storage/player.

## 9. Взаимодействие сайта и плеера

Плеер встраивается на страницу через `iframe`. Портал получает от storage/player готовые данные для встройки:

```text
iframeUrl
iframeHtml
manifestUrl
assetUrl
embedToken
query parameters
```

После загрузки iframe страница сайта и плеер могут обмениваться событиями через `window.postMessage`.

Примеры событий:

| Направление | Событие | Назначение |
|---|---|---|
| Сайт -> плеер | `viewer.load` | Загрузить представление или manifest. |
| Сайт -> плеер | `viewer.setCamera` | Перейти к нужной камере/точке интереса. |
| Сайт -> плеер | `viewer.openPoi` | Открыть точку интереса. |
| Сайт -> плеер | `viewer.setUi` | Включить/выключить панели, измерения, подписи. |
| Плеер -> сайт | `viewer.ready` | Плеер загрузился. |
| Плеер -> сайт | `viewer.poiClick` | Пользователь нажал точку интереса. |
| Плеер -> сайт | `viewer.cameraChange` | Изменилась камера. |
| Плеер -> сайт | `viewer.error` | Ошибка загрузки или доступа. |

`postMessage` API не заменяет REST API. REST API нужен для данных, прав, загрузок и embed. `postMessage` нужен для управления уже встроенным viewer на странице.

## 10. Как формируется публичная карточка и каталог

Публичная страница объекта собирается порталом:

```text
1. Портал получает объект по H3DID.
2. Проверяет статус и права доступа.
3. Подтягивает Dublin Core / CIDOC CRM / внешние ID.
4. Получает список twins и выбирает primaryTwinId.
5. Получает preview/screenshots через API сайта или storage.
6. Для основного представления получает embed от storage/player.
7. Рендерит страницу каталога или карточку объекта.
```

Storage/player не формирует культурную страницу каталога. Он только отдает технические ресурсы: preview, manifest, iframe, download URL и статусы обработки.

## 11. Минимальный набор API по слоям

API портала:

| Блок | Минимально нужно |
|---|---|
| Auth/session | Проверка пользователя и прав. |
| Objects | Создание и редактирование H3DID-объектов. |
| Metadata | Dublin Core, CIDOC CRM, внешние ID. |
| Organizations | Музеи, рабочие группы, роли. |
| Collections | Коллекции и подколлекции. |
| Twins | Цифровые представления объекта. |
| Assets | Связь сайта с файлами storage. |
| Catalog | Публичные списки объектов и коллекций. |
| Moderation | Черновики, review, публикация, архив. |

API storage/player:

| Блок | Минимально нужно |
|---|---|
| Uploads | Начать загрузку, завершить загрузку, вернуть ошибку лимита. |
| Assets | Получить технический статус и данные файла. |
| Processing | Статусы обработки и ошибки. |
| Preview | Скриншоты и превью для каталога. |
| Manifest | Техническое описание для viewer. |
| Embed | Сформировать iframe и query-параметры. |
| Download | Временные ссылки на скачивание. |
| Webhooks | Уведомления порталу о готовности/ошибках. |
| postMessage | Управление viewer внутри iframe. |

## 12. Итоговая схема

```text
Пользователь
  |
  v
Frontend Heritage3D
  |
  | user token
  v
API портала Heritage3D
  |
  | читает/пишет
  v
БД портала: H3DID, metadata, роли, коллекции, статусы
  |
  | service token
  v
Storage/player API
  |
  +--> S3: исходники, модели, tiles, preview, manifest
  +--> Processing: конвертация, оптимизация, генерация preview
  +--> Viewer/embed: iframe, manifest, assetUrl, embedToken

Публичная страница Heritage3D
  |
  v
iframe viewer
  |
  v
window.postMessage события между страницей и плеером
```

Коротко: портал управляет смыслом, правами и публикацией; storage/player управляет файлами, обработкой и просмотром; `H3DID` связывает культурный объект со всеми его цифровыми представлениями.
