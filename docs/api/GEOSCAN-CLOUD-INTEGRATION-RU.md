# Интеграция 3D-наследие с Геоскан Облаком

Этот документ фиксирует выводы из переписки по разработке и дополняет API-документацию портала.

## Главный вывод

Существующий frontend/модуль сторонней системы нельзя считать готовой переиспользуемой библиотекой для портала «3D-наследие». Для портала требуется отдельная бизнес-логика, свои схемы данных, API, статусы, роли, метаданные и сценарии модерации.

Оптимальная модель:

```text
Геоскан Облако
  identity provider
  хранение 3D-данных
  проекты и датасеты
  публикация/визуализация
  выдача embed-кода или embed-ссылки

3D-наследие
  портал и личный кабинет
  культурные объекты
  цифровые двойники
  CIDOC CRM / Dublin Core / отраслевые метаданные
  статусы и модерация
  связь с ЕГРОКН, Госкаталогом, музейными АИС
  связь с проектами Геоскан Облака
```

## Единая авторизация

По переписке реалистичная схема такая:

```text
Пользователь регистрируется / авторизуется через Геоскан Облако.
3D-наследие использует Геоскан Облако как identity provider.
3D-наследие выполняет запросы к API Геоскан Облака от имени пользователя.
```

Важно:

- учетная запись пользователя первично находится в Геоскан Облаке;
- нужно сохранить связь с realm `cloud`;
- если делать отдельный realm только для Heritage3D, может потеряться возможность выполнять API-запросы к Геоскан Облаку от лица пользователя;
- в 3D-наследие можно хранить собственный профиль пользователя и роли портала, но identity/authentication должны быть связаны с Геоскан Облаком.

## Хранение метаданных

Метаданные, которых нет в Геоскан Облаке, должны храниться в 3D-наследие:

- описание культурного объекта;
- Dublin Core;
- CIDOC CRM-связи;
- тип объекта: ОКН, музейный предмет, коллекция, фрагмент;
- дата съемки;
- метод съемки;
- точность;
- авторство;
- правообладатель;
- связи с ЕГРОКН, Госкаталогом, музейными АИС;
- статус модерации;
- доступ: public, unlisted, password, restricted, internal.

## Связь сущностей

В 3D-наследие нужно хранить связь с Геоскан Облаком:

```text
CulturalObject
  id: h3d:object:...

DigitalTwin
  id: h3d:twin:...
  objectId: h3d:object:...
  geoscanCloudProjectUuid: ...
  geoscanCloudDatasetUuid: ...
  geoscanCloudEmbedId / embedUrl: ...

MediaAsset
  id: h3d:asset:...
  storage.provider: geoscan_cloud или portal
  externalAssetId: ...
```

Для MVP возможны два варианта хранения:

```text
Вариант A:
  3D-модели физически хранятся в Геоскан Облаке.
  3D-наследие хранит метаданные и ссылки/UUID.

Вариант B:
  3D-модели хранятся на media-сервере портала.
  Геоскан Облако используется только для отдельных функций или не используется для хранения.
```

С учетом переписки вариант A выглядит реалистичным, если Геоскан Облако готово предоставить API для проектов, датасетов, загрузки и embed.

## Минимальный набор API Геоскан Облака

Для работы из интерфейса 3D-наследие нужен API Геоскан Облака минимум для:

```text
Создание проекта
Удаление проекта
Получение проекта
Список проектов пользователя

Создание / получение датасетов проекта
Загрузка файлов модели или исходных данных
Получение статуса обработки / публикации

Создание embed-кода
Получение embed-ссылки
Обновление параметров публикации
```

Желательно также:

```text
Проверка прав пользователя на проект
Получение списка доступных форматов
Получение preview/thumbnail
Получение статуса доступности viewer
Удаление или архивирование публикации
```

## Сценарий создания модели

```text
1. Пользователь авторизуется через Геоскан Облако.
2. В 3D-наследие открывает форму "Добавить 3D-модель".
3. Заполняет поля портала:
   - название;
   - описание;
   - тип культурного объекта;
   - дата съемки;
   - автор;
   - правообладатель;
   - идентификаторы ЕГРОКН/Госкаталог/АИС;
   - метод съемки;
   - параметры качества;
   - политика доступа.
4. 3D-наследие создает CulturalObject / DigitalTwin.
5. 3D-наследие через API создает проект в Геоскан Облаке от имени пользователя.
6. 3D-наследие сохраняет geoscanCloudProjectUuid.
7. Пользователь загружает модель или датасет.
8. 3D-наследие передает файл в связанный проект Геоскан Облака.
9. После публикации 3D-наследие получает embedUrl / embedCode.
10. 3D-наследие сохраняет embedUrl и формирует собственный manifest/embed API.
11. Объект проходит модерацию.
12. После публикации объект появляется в публичном каталоге портала.
```

## Сценарий получения embed

```text
1. Пользователь или публичная страница запрашивает POST /api/v1/twins/{twinId}/embed.
2. 3D-наследие проверяет статус, access policy и playerId.
3. Если модель хранится/публикуется через Геоскан Облако:
   - портал берет сохраненный embedUrl;
   - либо запрашивает актуальный embed через API Геоскан Облака;
   - сохраняет/обновляет ссылку.
4. Портал возвращает iframeUrl / iframeHtml.
```

## Что нужно запросить у команды Геоскан Облака

Для детального ТЗ нужно получить:

```text
1. Описание авторизации:
   - OAuth/OIDC endpoints;
   - realm;
   - scope;
   - refresh token;
   - как выполнять API-запросы от имени пользователя.

2. API проектов:
   - создать проект;
   - удалить проект;
   - получить проект;
   - права на проект.

3. API датасетов:
   - создать датасет;
   - загрузить файл;
   - получить статус;
   - удалить/архивировать.

4. API публикации:
   - создать публикацию;
   - получить embed;
   - обновить доступ;
   - снять с публикации.

5. Ограничения:
   - максимальный размер файла;
   - поддерживаемые форматы;
   - CORS;
   - срок жизни ссылок;
   - права на скачивание.
```

## Что нужно добавить в ТЗ 3D-наследие

```text
1. Роли пользователей портала.
2. Форму "Добавить 3D-модель".
3. Список обязательных метаданных.
4. Статусы объекта и цифрового двойника.
5. Политики доступа.
6. Связь DigitalTwin с geoscanCloudProjectUuid.
7. Сценарий загрузки файлов в Геоскан Облако.
8. Сценарий получения embed.
9. Сценарий модерации и публикации.
10. Что хранит 3D-наследие, а что хранит Геоскан Облако.
```

## Зона ответственности

```text
Геоскан Облако:
  - учетная запись / identity provider;
  - проекты;
  - датасеты;
  - хранение и публикация 3D-данных;
  - viewer/embed на стороне Облака;
  - API для загрузки и получения embed.

3D-наследие:
  - портал;
  - культурные объекты;
  - дополнительные метаданные;
  - модерация;
  - реестры и внешние идентификаторы;
  - собственный manifest/embed API;
  - пользовательский интерфейс для сценариев наследия.
```
